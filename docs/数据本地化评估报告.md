### 1.2 关键性能瓶颈分析

**单次分析的网络请求链路（以权益类基金为例）：**

```
用户输入基金代码
    ↓ (5-15s) fund_name_em() — 全量加载基金列表（~20000条）
    ↓ (2-5s)  fund_individual_basic_info_xq() — 基金基本信息
    ↓ (3-8s)  fund_open_fund_info_em() — 5年净值历史
    ↓ (2-5s)  fund_individual_detail_hold_xq() × 多季度 — 资产配置回退
    ↓ (2-5s)  fund_portfolio_hold_em() × 3年 — 股票持仓
    ↓ (2-5s)  fund_fee_em() — 费率
    ↓ (3-5s)  stock_zh_index_daily() — 基准指数
    ↓ (3-5s)  FF因子（5个指数 × stock_zh_index_daily）
    ↓ (3-8s)  全收益指数合成（get_total_return_series）
    ↓ (1-3s)  申万行业查询（如触发持仓穿透）
    ════════════════════════════════════════
    总计：26-69 秒（乐观估计）/ 更高（含重试）
```

**为什么经常需要5分钟：**
1. `fund_name_em()` 加载全量列表（~20000只基金）是主要瓶颈
2. 多个接口超时重试（每个最多3次，每次+2s延迟）
3. AkShare 数据源本身不稳定（东方财富/雪球/新浪）
4. Streamlit Cloud 网络环境到国内 API 延迟高
5. 转债类基金需要逐只查询详情（N次网络请求）

| 缓存类型 | 实现方式 | 范围 | 问题 |

**关键发现：`@cached()` 装饰器是空壳！** 这意味着所有标注了 `@cached` 的函数实际上没有任何缓存效果，每次都会重新调用 API。

### 1.5 Engine 层审计结论

> 审计 `engine/` 全部 6 个 Python 文件：`equity_engine.py`, `bond_engine.py`, `index_engine.py`, `cb_engine.py`, `common_metrics.py`, `__init__.py`

**结论：Engine 层无任何直接外部 API 调用。** 所有 engine 模块只接收 data_loader 传入的数据进行纯计算（CAPM、Brinson 归因、久期估算等），不存在独立的外部数据依赖。数据本地化只需改造 `data_loader/` 层，engine 层零改动。

### 3.2 存储技术选型对比

| 方案 | 适用场景 | 优点 | 缺点 |
|---|---|---|---|
| **Parquet文件** | 时间序列（净值/指数） | 压缩好、读取快、pandas原生 | 不支持增量写入 |
| **SQLite** | 结构化数据（基金信息/持仓） | 零运维、单文件、SQL查询 | 并发写入能力弱 |
| **PostgreSQL** | 生产环境 | 功能完整、高并发 | 需要运维 |
| **Redis** | 热数据缓存 | 极快 | 内存贵、不持久 |

**推荐组合**：SQLite（基金信息+持仓） + Parquet（净值+指数） = **最佳性价比**

## 四、问题三：云端存储 + Streamlit Cloud 全流程方案

### 4.1 Streamlit Cloud 的关键限制

| 限制项 | 具体限制 | 影响 |
|---|---|---|
| **无持久化存储** | 每次部署/重启后文件系统被清空 | ❌ 本地 Parquet/SQLite 会被删除 |
| **内存限制** | Community 免费：1GB RAM | 可用，但需控制数据加载量 |
| **CPU限制** | Community 免费：共享CPU | 数据加载可以，不宜做繁重计算 |
| **执行时间** | 无硬性限制，但超时会断开 | 需在60秒内完成 |
| **无后台进程** | 无法运行 cron/定时任务 | ❌ 不能自动更新数据 |
| **GitHub集成** | 自动部署 | ✅ 代码更新方便 |

### 4.3 方案详细设计

#### 🌟 推荐方案：Supabase + GitHub Actions（总成本 ≈ 0 元）

**为什么推荐 Supabase：**
- **免费额度足够**：500MB 数据库 + 1GB 文件存储 + 50,000 月活行读取
- **PostgreSQL 原生**：支持 JSONB、时间序列、全文搜索
- **自带 REST API**：Streamlit 直接通过 HTTP 调用，无需驱动
- **全球 CDN**：数据读取延迟低
- **无需运维**：自动备份、自动扩缩

**全流程：**

```
步骤1: 数据初始化（一次性）
├── 本地脚本：通过 AkShare 批量拉取 20,000 只基金数据
├── 写入 Supabase PostgreSQL 数据库
└── 预计耗时：4-8 小时（受 API 限速影响）

步骤2: 数据更新（每日自动）
├── GitHub Actions cron: 每天 18:00 CST 触发
├── 增量更新净值数据（只拉当天新增）
├── 检查季报/年报持仓更新
├── 更新指数数据
└── 预计耗时：30-60 分钟

步骤3: 用户分析（实时）
├── Streamlit Cloud 接收用户输入
├── Layer 1: 检查内存缓存（5分钟TTL）→ 命中则秒回
├── Layer 2: 未命中 → 查询 Supabase（网络~100ms）
├── Layer 3: 数据库也无 → 回退 AkShare API + 异步写入数据库
└── 分析计算 → 返回报告

步骤4: 代码部署
├── 修改 data_loader 层：优先数据库 → 回退 API
├── 新增 data_loader/cloud_db.py：Supabase 客户端
├── GitHub push → Streamlit Cloud 自动部署
└── 环境变量：SUPABASE_URL, SUPABASE_KEY
```

## 23:45 - 修复债券基金类型识别错误（000069 测试案例）

### 问题描述
用户测试债券基金 000069 时发现报告中有"选股"等不合适的表述。

### 根本原因
1. `load_basic_info()` 返回的 `type_category` 是中文类型（"纯债型"、"股票型"等）
2. `pipeline.py` 中的类型判断逻辑使用英文类型码（"bond", "equity", "mixed"等）
3. 中文类型无法匹配英文类型码，导致所有未知类型被 fallback 到权益分析
4. 权益分析使用 `equity_report_writer.py`，其中包含"选股"等表述

### 修复方案
在 `pipeline.py` 中添加类型映射函数 `_map_type_category()`：

```python
def _map_type_category(chinese_type: str) -> str:
    """将中文类型分类映射到英文类型码"""
    # 债券类
    if chinese_type in ("纯债型", "中短债", "长债", "债券型"):
        return "bond"
    # 股票/指数类
    if chinese_type in ("股票型", "增强指数"):
        return "equity"
    # 混合类
    if chinese_type in ("偏股混合型", "偏债混合型", "平衡混合型", "混合型"):
        return "mixed"
    # ... 其他类型
```

## 17:41 - 超额收益逻辑差异分析与结论

### 问题分析
关于"超额收益逻辑差异(3.33e-05)"的问题，经过深入分析发现：

**根本原因**：该差异源于 `tests/functional_consistency_check.py` 中测试用例的数据问题。

测试调用：`check_excess_return_logic(0.25, 0.20, 0.0417)`

- 基金收益: 25%
- 基准收益: 20%
- 代码超额: 4.17% (这是四舍五入后的值)

**手动精确计算**：
```
超额收益 = (1 + 基金收益) / (1 + 基准收益) - 1
        = (1 + 0.25) / (1 + 0.20) - 1
        = 1.25 / 1.20 - 1
        = 1.0416667 - 1
        = 0.0416667 (即 4.1667%)
```

**差异来源**：
测试用例中传入的 `0.0417` 是四舍五入后的近似值，而精确值应为 `0.0416667`，两者差异约为 `0.0416667 - 0.0417 ≈ -0.0000333 = -3.33e-05`。

### 结论
1. **这不是算法问题**：几何超额计算逻辑本身是正确的
2. **这是测试数据问题**：应该使用精确值 `0.0416667` 而不是近似值 `0.0417`
3. **不需要调整容差**：调整容差 1e-6 → 1e-4 会掩盖测试数据的精度问题
4. **正确做法**：修复测试用例，使用精确值

### 建议
- 将 `check_excess_return_logic(0.25, 0.20, 0.0417)` 改为 `check_excess_return_logic(0.25, 0.20, 0.0416667)`
- 或使用更精确的浮点数表示，避免四舍五入引入的精度损失

### 关于统一精度常量的补充说明
如 MEMORY.md 中所述，虽然差异不是算法问题，但建议在 `engine/common_metrics.py` 中定义：
```python
FINANCIAL_EPSILON = 1e-4  # 1个基点(1BP)的1/100，金融工程中的最大允许误差
```
这可以作为系统的"最大允许误差"标准，而 1e-6 继续用于严格的数值相等性检查。

## v9.4 数据管线全面重构（10项关键修复）

**文件：`fund_analysis.py` = `fund_analysis_v9.4.py`**

### 修复1（致命）：Alpha量纲Bug
- **原因**：`_run_single_ff` 用标准化后的X回归截距作为Alpha，量纲严重失真（截距≠Alpha）
- **修复**：Alpha从**原始回归（m_raw）**截距取，标准化回归只用于横向Beta比较，Alpha不再为0

### 修复2（致命）：R²变量作用域Bug
- **原因**：`run_ff_model` 中R²突降检测（第992行）引用了`r2`变量，但该变量要到第1014行才赋值→NameError或用错值
- **修复**：提前从`res_full`取出所有结果变量，R²突降检测移到正确位置

### 修复3（高风险）：数据对齐"不满60天"根因
- **原因**：`run_ff_model`和`run_duration_model`都用`inner join + dropna()`，因子/国债数据有轻微缺口就会大量丢行
- **修复**：改为`left join（以基金净值为准）+ ffill(limit=3) + dropna(subset=核心列)`，错误信息加上原始行数/对齐后行数辅助诊断

### 修复4（高风险）：久期"忽零忽五"
- **原因1**：R²<0.3时回归不显著，但代码仍用噪音久期进压力测试
- **原因2**：数据不足时默认值硬编码5.0（偏大）
- **修复**：R²<0.3→回退到行业经验值2.5年（`fallback_low_r2`）；数据不足默认2.5年；加np.clip(0.1, 15.0)范围保护；压测展示加"（经验估算值）"标注

### 修复5（高风险）：基准变横线
- **原因**：`fetch_index_daily`的`pct_change()`首行永远是NaN，`dropna()`后第一行收益率丢失，`cumprod`时出现传播问题
- **修复**：`pct_change().fillna(0)`，首行填0不影响累计收益从1出发；`build_benchmark_ret`各出口都加`fillna(0)`防护

### 修复6：跨市场估值路由（港股/QDII）
- **修复**：加`_identify_market()`识别A股(6位)/港股(5位)/其他；港股用`stock_hk_valuation_baidu`；OTHER类标注"跨市场个股，估值数据暂缺"

### 修复7：残差UI优化

- 残差>0.5%时在饼图下方加说明卡，解释打新/日内交易等不可分解的合理超额

## v9.5 全面代码审查修复（8项潜在Bug）

**文件：`fund_analysis.py` = `fund_analysis_v9.5.py`**

### Bug1（高风险）：bond模式translate_results/calc_radar_scores数据路径错误
- **根因**：bond模式下`model_results={'bond': bond_res}`，但translate_results/calc_radar_scores直接`results.get('duration')`取不到数据，全部返回默认值
- **修复**：调用前检测bond模式并展开子字典：`_translate_results = model_results['bond']`

### Bug2（高风险）：fetch_ff_factors内部inner join截断因子数据
- **根因**：构建FF因子矩阵时，small/val/grw与mkt做inner join，某个指数停牌一天会截断所有行
- **修复**：改为left join + ffill(limit=3)，以沪深300日期为基准

### Bug3（中风险）：_empty_ff_result缺少r_squared_recent和residual_insight字段
- **根因**：展示层读这两个字段，但_empty_ff_result未返回，导致KeyError或AttributeError
- **修复**：补全两个字段（None/''}）

### Bug4（中风险）：estimate_hidden_cost的type_category永远是equity
- **根因**：调用时未传入type_category，函数签名无此参数，所有基金都用equity隐性费率0.15%
- **修复**：函数签名加type_category参数，调用处传入`basic['type_category']`

### Bug5（中风险）：run_brinson使用dir()判断变量存在（Python反模式）
- **根因**：`r_bm_ann if 'r_bm_ann' in dir() else 0.0`，dir()是类方法列表，不是本地变量检测
- **修复**：改为直接引用（两条分支均保证变量被赋值）

### Bug6（中风险）：fetch_bond_index首行NaN未填0
- **根因**：与fetch_index_daily不一致，bond_index用dropna()丢掉了首行
- **修复**：pct_change().fillna(0)，去掉dropna()

### Bug7（低风险）：sector基金估值预警右列在bond_ratio<=0.10时不渲染
- **根因**：_has_right_content条件未检查top10是否有数据
- **修复**：加持仓数据非空检测

### Bug8（低风险）：calc_radar_scores的bond模式也有嵌套字典问题
- **根因**：与Bug1同源，calc_radar_scores传入{'bond':bond_res}
- **修复**：同样展开bond子字典

## 显性费率永远为0的根本原因修复

**问题**：费率卡片显示管理费/托管费/显性合计均为0
**根因**：
1. 雪球接口 `fund_individual_basic_info_xq` 返回的字段里根本没有「管理费率」「托管费率」「销售服务费率」（实测14个字段只有基金基本信息）
2. 天天基金 `fund_open_fund_info_em(indicator='基金概况')` 返回空 DataFrame（indicator参数不支持此值）
3. 两层数据源都取不到费率，所以 fee_manage/fee_custody/fee_sale 永远是初始值 0.0

**修复**：
- 主方案：改用 `ak.fund_fee_em(symbol, indicator='运作费用')`，返回格式为固定列序：col1=管理费率值, col3=托管费率值, col5=销售服务费率值
- 兜底方案：`ak.fund_individual_detail_info_xq(symbol)` 读「其他费用」行，含「基金管理费」和「基金托管费」（值为 float 百分比数字，加%后交_parse_fee解析）
- 实测：000001 管理费1.20% 托管费0.20% 合计1.40%；000961 管理费0.50% 托管费0.10% 合计0.60%，读取正确

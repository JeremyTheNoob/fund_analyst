```python
def _ak_fund_list_em() -> Optional[pd.DataFrame]:
    """
    获取基金代码和名称列表（东方财富）
    
    Returns:
        包含基金代码和名称的DataFrame，列名为['基金代码', '基金名称']
        如果没有数据返回None
    """
    try:
        df = safe_api_call(
            lambda: ak.fund_name_em()
        )
        if df is not None and not df.empty:
            if '基金代码' in df.columns and '基金名称' in df.columns:
                return df[['基金代码', '基金名称']].copy()
            elif len(df.columns) >= 2:
                df.columns = ['基金代码', '基金名称'] + list(df.columns[2:])
                return df[['基金代码', '基金名称']].copy()
    except Exception as e:
        logger.warning(f"_ak_fund_list_em 失败: {e}")
    return None
```

## 申万指数全收益计算功能重构完成（03:36-04:15）

### **问题背景**
用户反馈在测试 `test_total_return_integration.py` 时发现申万行业指数数据获取失败：
- 原代码使用 `PriceIndexFetcher.fetch_price_series()` 调用不存在的接口
- 申万行业指数需要专门的数据源支持
- 需要实现"一石二鸟"的工程最优解：在一个请求中同时获取价格和股息率

### **解决方案分析**

#### **数据源选择决策**
经过详细分析，确定了两个关键申万指数接口：

1. **`index_analysis_daily_sw` 接口**（选用）：
   - **功能**：申万指数分析日报表
   - **优势**：同时提供收盘指数（价格）和股息率
   - **股息率类型**：**TTM股息率**（滚动计算，每日更新）
   - **返回数据**：所有指数的当日分析数据
   - **调用方式**：`ak.index_analysis_daily_sw(symbol="一级行业", start_date="YYYYMMDD", end_date="YYYYMMDD")`

2. **`sw_index_first_info` 接口**（参考）：
   - **功能**：申万一级行业基本信息
   - **股息率类型**：**静态股息率**（基于最新年报）
   - **数据更新**：每年4月30日年报披露后更新
   - **用途**：作为固定股息率参数的参考来源

#### **技术决策**：选择TTM股息率的原因
1. **准确性**：TTM股息率滚动计算过去12个月的分红，更准确反映实际分红收益
2. **实时性**：每日更新，能及时反映市场变化
3. **一致性**：与价格数据同源，确保数据时间对齐
4. **工程效率**：一个接口同时获取价格和股息率，减少API调用

## 累计收益曲线三个关键错误修复（15:20-15:30）

### **问题诊断**
通过深入分析发现了累计收益曲线的三个潜伏错误：

#### **1. 计算逻辑冲突（基准不一）**
- **错误**：代码中使用`bm_ret`（价格收益率），但数据标准要求"全收益优先"
- **后果**：如果使用价格指数计算基准累计收益，会漏掉分红，导致基准线偏低，虚增基金经理Alpha
- **发现**：`_cumulative_return_chart()`函数第177行直接使用`bm_df['bm_ret']`，没有检查是否全收益数据

#### **2. 归一化起点错误（利滚利偏差）**
- **错误**：`cum_fund = (1 + ret).cumprod() - 1`
- **后果**：如果ret序列的第一天不是0，直接cumprod会导致起始点跳空，基金与基准在t0时刻没有同时从0%出发
- **发现**：代码没有强制零点对齐，可能导致基金与基准起点不一致

#### **3. 频率对齐导致的"平仓幻觉"**
- **错误**：仅使用前向填充`reindex(method='ffill')`
- **后果**：如果基金在长假前净值未更新，而基准有数据，前向填充会产生水平线，误导AI解读为"经理主动减仓避险"
- **发现**：没有数据有效性检查，可能导致数据失真

### **修复方案实现**

#### **1. 统一基准算力**
```python
# 优先使用全收益数据
target_ret_col = 'tr_ret' if 'tr_ret' in bm_df.columns else 'bm_ret'
```

#### **2. 强制零点对齐**
```python
# 确保序列第一行为0，保证曲线从(0,0)点出发
nav_df['ret_clean'] = nav_df['ret'].fillna(0)
nav_df['cum_fund'] = (1 + nav_df['ret_clean']).cumprod() - 1

# 确保第一个累计收益为0
if nav_df['cum_fund'].iloc[0] != 0:
    nav_df['cum_fund'] = nav_df['cum_fund'] - nav_df['cum_fund'].iloc[0]
```

```python
# 识别连续不变的基准收益率（可能表示数据缺失）
zero_streak = (bm_series == 0).astype(int)
```

## 水下回撤图三个关键错误修复（15:36-15:45）

### **问题诊断**
通过深入分析发现了水下回撤图的三个潜伏错误：

#### **1. 基准选择的"苹果比橘子"错误**
- **错误**：计算过程使用`bm_ret`（价格收益率），但基础数据中基准只有价格收益率，没有包含分红的`tr_ret`
- **后果**：由于基准没有包含分红，价格指数在长期会显得比全收益指数"更弱"。如果用一个不含分红的基准算回撤，基准回撤会很大（因为价格跌得深），从而虚增了基金的防御能力
- **发现**：`_drawdown_chart()`函数第281行直接使用`bm_df['bm_ret']`，没有优先使用全收益数据

#### **2. 起始点"虚假回撤"风险**
- **错误**：`cum_fund = (1 + ret).cumprod()`
- **后果**：如果ret序列是从基金成立后的某一天截取的（比如回测近三年），此时的cum_fund起始值是1。但如果该基金之前在历史高位，现在的1其实已经是在回撤中了
- **发现**：没有区分绝对回撤（基于历史最高点）和区间回撤（基于当前显示窗口的最高点）

#### **3. 恢复期的"假死"逻辑**
- **错误**：未定义"回撤结束"的阈值
- **后果**：在计算"修复天数"时，如果净值只差0.0001没创新高，逻辑上它仍处于回撤中。这会导致AI误判经理"长期无法回本"，而实际上可能只是震荡微跌
- **发现**：没有修复阈值和关键点标记机制

### **修复方案实现**

#### **1. 基准对齐（必须使用全收益）**
```python
# 优先使用tr_ret（全收益收益率），因为分红可以垫高净值，缩小回撤深度
target_ret_col = 'tr_ret' if 'tr_ret' in bm_df.columns else 'bm_ret'
```

#### **2. 引入"高水位线"清洗**
```python

epsilon = 1e-10
nav_df['drawdown_fund'] = (nav_df['cum_fund'] / (cum_max + epsilon)) - 1
```

#### **3. 增加"回撤修复"关键点标记**

## 月度热力图三个关键错误修复与自动化解读模板集成（15:47-16:15）

### **问题诊断**
通过深入分析发现了月度热力图的三个潜伏错误：

#### **1. "首尾月"计算偏差**
- **错误**：直接按 `(1 + ret).prod() - 1` 计算月度收益率
- **后果**：如果数据开始于某月15日，或者结束于某月10日，该月的收益率实际上只有"半个月"。在热力图中，这半个月的涨跌会与完整的自然月并列显示，导致误判经理在该月的择时能力
- **发现**：代码没有检查有效交易日数，首尾月被当作完整月计算

#### **2. 年度收益（Year-to-Date）缺失**
- **错误**：仅进行了 `unstack()` 矩阵转换

- **发现**：月度矩阵缺少年度汇总列

#### **3. 算术平均 vs 几何平均的混淆**
- **错误**：未正确处理NaN值
- **后果**：如果某月因停牌导致全月无数据，`prod()` 的结果可能是1（即收益率为0%），这会误导用户认为该月"平盘"，而实际应标记为"无数据"或"停牌"

## 超额收益曲线三个关键错误修复与优化（17:01-17:25）

### **问题诊断**
通过深入分析发现了超额收益曲线的三个关键错误：

#### **1. 计算方法的"算术陷阱"（Arithmetic vs Geometric）**
- **错误**：原代码使用 `excess_return = cum_fund - cum_bm`（减法逻辑）
- **后果**：这是新手最容易犯的错。在复利效应下，简单相减会由于基数不同产生巨大的数值偏差
- **案例**：假设基准跌了-50%，基金跌了-40%。按算术公式超额是10%。但实际上，你拿60块钱去追50块钱的基准，你的资产比基准多了20%（60/50−1）

#### **2. 全收益基准缺失导致的"虚假Alpha"**
- **错误**：基础数据中仅使用 `bm_ret`（价格收益率）
- **后果**：由于未包含分红，基准线被低估。在高分红行业（如银行、煤炭），你会发现超额曲线一直斜向上，其实那不是经理厉害，而是你忘了算基准的分红

#### **3. 起始点对齐漂移**
- **错误**：没有强制 `excess_curve[0] = 0`
- **后果**：如果基金和基准的初始日期由于ffill产生了一天的错位，第一天的超额就不为0，导致整条曲线像是在"悬空"飞行

### **修复方案实现**

#### **1. 改用"几何超额"算法（专业标准）**
```python
# 专业标准：excess_curve = (fund_nav / bm_nav) - 1
# 这反映"1块钱投入基金比投入基准多赚了多少"

```

#### **2. 强制全收益对齐**
```python

target_ret_col = None
if 'tr_ret' in bm_df.columns:
    target_ret_col = 'tr_ret'
    ret_type = '全收益'
elif 'bm_ret' in bm_df.columns:
    target_ret_col = 'bm_ret'
    ret_type = '价格收益'
```

#### **3. 净值起点强制对齐**
